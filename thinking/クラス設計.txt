前提として、glfw, opengl/3hのライブラリはCで書かれている。
これらをクラスとして管理することで、可読性を上げるようにしてあげたい。
そのために、どういう風にクラス設計をすればよいか。
これらのライブラリの性質+今回達成したいゲームの機能、実装+そもそもゲーム開発におけるセオリー
をもとに考える


1. まずは、今回の課題を達成するために必要な実装を日本語でまとめる
2. それをもとに実際のコードに当てはめる
3. クラス設計に落とし込む

短く言うと：クラス化は多くの場合「ベストに近い妥当解」だが、必須ではなく要件次第で最適解が変わります。

- いつクラス化が有利か（メリット）
  - 長期保守／拡張予定がある：責務分離と再利用で作業が楽になる。  
  - 複数ウィンドウや複数レンダラを扱う／リソース管理が増える：RAIIで確実に解放できる。  
  - 単体テストや差し替え（モック化）をしたい：インタフェース化しやすい。  

- いつクラス化は過剰か（デメリット）
  - 単純なプロトタイプや一発デモ：手間増で得るものが少ない。  
  - 極端なパフォーマンスが最優先で関数呼び出しオーバーヘッドを最小化したい場合（通常は無視できるレベル）。  

- 他の選択肢
  - 完全手続き型（C風）: 素早く簡潔。小さな実験向け。  
  - 最小ラッパー（関数＋1つのstruct/opaque）: C API とほぼ同じだが所有権だけRAIIにする折衷案。  
  - ECS/データ指向（大規模ゲーム向け）: 入力/描画/更新を分離して高性能化。設計コスト高。  
  - 既存の軽量フレームワークを使う（glfw+gladラッパー等）: 自前実装コストを削減。

- 判断基準（あなたに合うか決める方法）
  1. プロジェクト寿命：短期→手続き／長期→クラス化  
  2. 複雑さ：複数モジュール／サブシステムがある→クラス化推奨  
  3. テスト性や再利用性が重要か→クラス化推奨  
  4. 素早く動く試作を優先→手続きまたは最小ラッパー

推奨（あなたの repo がレンダリング系で長く触る想定なら）：
- 最小限のクラス群（`Application`, `Window`, `Renderer`, `ResourceManager`）を導入。まずは `Window` + コールバック転送 と `Renderer` の最小実装から始めると投資対効果が良い。
